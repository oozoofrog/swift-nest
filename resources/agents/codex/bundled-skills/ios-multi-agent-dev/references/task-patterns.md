# Task Patterns

작업 유형에 따라 전체 흐름을 그대로 쓰거나 압축하기.
단, 구현과 리뷰 역할 분리, 메인 에이전트 검증 루프는 유지하기.

## 기능 추가

권장 순서:
1. Planner/Reviewer 문제 정의
2. Builder 분석
3. Builder 구현
4. 메인 에이전트 검증
5. Planner/Reviewer 리뷰
6. Builder 수정 반영
7. 메인 에이전트 재검증

적용 기준:
- UI/상태 흐름이 추가되기
- API 또는 저장소 연동이 들어가기
- 사용자 플로우가 늘어나기

## 버그 수정

권장 순서:
1. Builder 원인 후보 분석
2. Planner/Reviewer 재현 조건과 회귀 전략 정리
3. Builder 수정
4. 메인 에이전트 검증
5. Planner/Reviewer 검증 리뷰
6. 필요 시 Builder 재수정
7. 메인 에이전트 재검증

적용 기준:
- 증상보다 원인을 먼저 특정해야 하기
- 재현 조건이 불명확하기
- 회귀 우려가 높은 핫픽스이기

## 대규모 리팩터링

권장 순서:
1. Planner/Reviewer 단계 분할
2. Builder 1단계 분석
3. Builder 1단계 구현
4. 메인 에이전트 검증
5. Planner/Reviewer 단계 리뷰
6. 다음 단계 반복

규칙:
- 한 번에 한 단계만 수정하기
- 각 단계마다 컴파일 가능 상태 유지하기
- 공개 API 변경은 단계별로 분리하기

## 릴리즈 직전 변경

권장 순서:
1. Planner/Reviewer가 실패 기준과 롤백 조건 먼저 정의하기
2. Builder 분석에서 수정 범위를 최대한 줄이기
3. Builder 구현 후 메인 에이전트가 직접 검증 명령 실행하기
4. Planner/Reviewer가 릴리즈 리스크만 집중 리뷰하기
5. 필요 시 Builder 수정 후 메인 에이전트 재검증하기

규칙:
- 과도한 정리 금지
- 로그/플래그/가드 추가는 허용하되 동작 변경은 최소화하기
- 테스트가 없으면 수동 QA 체크리스트를 반드시 남기기

## 단순 변경 압축하기

아래 조건을 모두 만족하면 분석과 구현 단계를 한 worker 흐름으로 압축할 수 있다.
- 수정 파일 수가 매우 적기
- 상태 관리/동시성/네트워크 영향이 작기
- 공개 API 변화가 없기
- 실패해도 롤백이 쉬우기

그래도 다음은 유지하기.
- 메인 에이전트의 Task Brief 정리
- 메인 에이전트의 구현 검증
- 독립 리뷰
