# Review Checklist

리뷰 단계에서는 칭찬보다 결함 탐지를 우선하기.
구현자 관점이 아니라 릴리즈 리스크 관점에서 보기.

## 형식 준수 게이트

다음 중 하나라도 해당하면 리뷰를 무효로 보고 다시 요청하기.
- 요청한 7개 헤딩을 그대로 쓰지 않음
- 구현 요약이나 칭찬이 대부분을 차지함
- 문제 지적에 파일/함수/근거가 없음
- 수정 요청이 추상적이라 바로 반영할 수 없음
- 비어 있는 섹션을 넘어감

재요청할 때 강제하기.
- 각 이슈마다 `어디서`, `왜`, `영향`, `최소 수정 방향`을 쓰기
- 이슈가 없으면 `없음`을 명시하기
- 장점이나 요약은 한 문장 이하로 제한하기

## 우선순위

1. 크래시 가능성
2. 데이터 무결성 문제
3. Swift Concurrency 오류
4. UI 상태 불일치
5. 생명주기 이슈
6. 메모리 누수 및 retain cycle
7. 네트워크 실패 흐름 누락
8. 테스트 누락

## 치명적 문제로 분류하기

다음이 보이면 최우선으로 올리기.
- 강제 언래핑, 인덱스 접근, 옵셔널 처리 실수
- MainActor 위반으로 UI가 비결정적으로 깨질 가능성
- 중복 요청이나 중복 저장으로 데이터가 오염될 가능성
- observer/delegate 해제 누락으로 중복 이벤트 또는 누수 발생 가능성
- 화면 재진입/빠른 탭에서 상태가 꼬일 가능성

## 높은 확률의 버그로 분류하기

- 빈 상태, 로딩 상태, 실패 상태 중 하나가 빠짐
- 성공 흐름만 있고 실패/취소 흐름이 없음
- 재시도 후 상태 초기화가 누락됨
- 테스트 더블 또는 스텁이 실제 동작과 어긋남
- 버그를 가리는 guard만 추가하고 원인은 남겨둠
- SwiftUI에서 `.task` 재실행과 `.alert` dismiss 흐름이 불명확함

## 수동 QA 체크리스트

- 빠르게 두 번 이상 탭하기
- 화면 진입 직후 뒤로 가기
- 네트워크 느림/실패 환경에서 재시도하기
- 앱 백그라운드/포그라운드 전환하기
- 로그아웃/재로그인 또는 세션 만료 상황 재현하기
- 긴 리스트에서 재사용 셀 스크롤 확인하기
- 앱 재실행 후 상태 복원 확인하기
- SwiftUI alert를 띄운 뒤 dismiss 후 상태 초기화 확인하기

## 리뷰 산출물 형식

다음을 분리해 보고하기.
- 반드시 수정해야 하는 문제
- 릴리즈 전 확인이 필요한 위험
- 후속 개선으로 미뤄도 되는 항목
- 추가 테스트가 필요한 항목
